# AgentGateway 后端服务接入指南

## 你需要做什么

Gateway + OPA 已经帮你做完了所有认证授权工作（JWT 验签、角色校验、租户隔离）。你的服务只需要：

1. 按路径规则设计 API
2. 从 `X-Auth-*` Headers 读取用户信息（OPA 授权通过后自动注入）

---

## 第 1 步：按规则设计 API 路径

你的 API 路径必须符合以下两种模式之一，OPA 会根据路径自动判断所需权限：

```
普通用户接口:   /{tenant-id}/{app}/{resource}
管理员接口:     /{tenant-id}/{app}/admin/{resource}
```

**关键点：有没有 `/admin/` 这一段决定了权限级别。**

### 示例

假设你的应用叫 `da`，租户是 `data-agent`：

| 你的 API | 权限级别 | 谁能访问 |
|---------|---------|---------|
| `/data-agent/da/patients` | 普通用户 | 所有登录用户 |
| `/data-agent/da/records/detail/123` | 普通用户 | 所有登录用户 |
| `/data-agent/da/admin/settings` | 租户管理员 | `tenant_admin` 及以上 |
| `/data-agent/da/admin/users` | 租户管理员 | `tenant_admin` 及以上 |
> `{tenant-id}` 和 `{app}` 都可以自定义，只要遵守格式。OPA 自动从路径提取 tenant-id 并与 Token 中的 realm 做隔离校验。
> 如果还需要超管接口（跨租户），路径为 `/super-admin/{app}/{resource}`，需要 `super_admin` 角色。

---

## 第 2 步：读取用户信息

请求到达你的服务时，OPA 已经验证完毕并注入了以下 Headers：

| Header | 内容 | 示例 |
|--------|------|------|
| `X-Auth-User-Id` | 用户唯一 ID | `f47ac10b-58cc-4372-...` |
| `X-Auth-Username` | 用户名 | `g-5e3f01eb-...` |
| `X-Auth-Roles` | 角色（逗号分隔） | `tenant_admin,default-roles-data-agent` |
| `X-Auth-Email` | 邮箱 | `user@example.com` |
| `X-Auth-Tenant` | 租户标识 | `data-agent` |

---

## 第 3 步：部署并接入 Gateway

### 3.1 部署你的服务

正常部署 Deployment + Service，假设：
- namespace: `your-ns`
- Service 名: `da-api`
- 端口: `8080`

### 3.2 创建 ReferenceGrant（允许跨 namespace 路由）

```yaml
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-routes-to-da
  namespace: your-ns                    # ← 你的 namespace
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: agentgateway-system
  to:
  - group: ""
    kind: Service
    name: da-api                        # ← 你的 Service 名
```

### 3.3 创建 HTTPRoute

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: da-api-route
  namespace: agentgateway-system
spec:
  parentRefs:
  - name: agentgateway-proxy
    namespace: agentgateway-system
  hostnames:
  - "localhost"                         # ← 生产环境改为实际域名
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /data-agent/da           # ← 你的路径前缀
    backendRefs:
    - name: da-api                      # ← 你的 Service 名
      namespace: your-ns                # ← 你的 namespace
      port: 8080                        # ← 你的端口
```

> **注意**：如果你想保留原始路径，不需要加 URLRewrite filter。

### 3.4 挂载 ext_authz 策略

```yaml
apiVersion: agentgateway.dev/v1alpha1
kind: AgentgatewayPolicy
metadata:
  name: da-api-auth-policy
  namespace: agentgateway-system
spec:
  targetRefs:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    name: da-api-route                  # ← 与上面 HTTPRoute 同名
  traffic:
    extAuth:
      backendRef:
        name: opa
        namespace: agentgateway-system
        port: 9191
      grpc: {}
```

**这一步挂上之后，所有到你服务的请求都会先经过 OPA 认证授权。**

---

## 第 4 步：验证

```bash
# 获取 Token
ACCESS_TOKEN=$(curl -s -X POST \
  "http://localhost/realms/data-agent/protocol/openid-connect/token" \
  -d "grant_type=password" \
  -d "client_id=xxxx" \
  -d "client_secret=xxxx" \
  -d "username=xxxx" \
  -d "password=xxxx" | jq -r '.access_token')

# ✅ 带 Token 访问 — 应返回 200 + 你的业务响应
curl -i http://localhost/data-agent/da/patients \
  -H "Authorization: Bearer $ACCESS_TOKEN"

# ❌ 不带 Token — 应返回 403
curl -i http://localhost/data-agent/da/patients

# ❌ 伪造 Token — 应返回 403
curl -i http://localhost/data-agent/da/patients \
  -H "Authorization: Bearer fake.token.here"
```

## FAQ

**Q: 我的服务收不到 X-Auth-* Headers？**

检查：AgentgatewayPolicy 是否创建成功；Token 是否有效（默认 5 分钟过期）。

**Q: 路径前缀可以自定义吗？**

`{tenant-id}` 和 `{app}` 都可以自定义，只要遵守格式。OPA 自动从路径提取 tenant-id 并与 Token 中的 realm 做隔离校验。

**Q: 多个服务怎么接入？**

每个服务重复第 3 步（ReferenceGrant + HTTPRoute + AgentgatewayPolicy），用不同的路径前缀区分即可。
